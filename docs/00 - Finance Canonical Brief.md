# Finance Canonical Brief v1

## 1. Σκοπός του εγγράφου
 
Το παρόν έγγραφο αποτελεί το θεμέλιο αναφοράς για το Finance System v1. Ορίζει την ταυτότητα, τη σκοπιμότητα, τα όρια και τη θεμελιώδη λογική του συστήματος. Λειτουργεί ως ο «καταστατικός χάρτης» που διασφαλίζει την ενιαία ερμηνεία των εννοιών, ανεξάρτητα από την τεχνική υλοποίηση ή το UI.  
  
Η πιο λεπτομερής ευθυγράμμιση του domain model ορίζεται στο `00A — Finance Domain Model & System Alignment`.

## 2. Σκοπός του συστήματος

Το Finance System v1 σχεδιάστηκε για να παρέχει λειτουργική ορατότητα στην καθημερινή οικονομική δραστηριότητα της επιχείρησης. Στόχος είναι η δημιουργία μιας «κοινής γλώσσας» που επιτρέπει στην ομάδα να γνωρίζει ανά πάσα στιγμή:

**Έσοδα**: Τι αναμένεται, τι έχει τιμολογηθεί και τι απομένει προς είσπραξη.
**Δαπάνες**: Τι έχει ζητηθεί, τι έχει εγκριθεί (δεσμευτεί) και τι πρέπει να πληρωθεί.
**Διαχείριση**: Ποια σημεία απαιτούν άμεση παρέμβαση (σήματα ελέγχου).

## 3. Scope του v1

Το σύστημα διαρθρώνεται σε δύο λειτουργικούς κύκλους και ένα επίπεδο εποπτείας:

### Α. Κύκλος Εσόδων (Revenue Chain)
Billable Work -> Invoice Draft -> Invoice -> Receivable -> Incoming Payment

### Β. Κύκλος Δαπανών (Spend Chain)
Purchase Request -> Commitment -> Supplier Bill -> Outgoing Payment

### Γ. Επίπεδο Εποπτείας & Ελέγχου (Monitoring & Control)
Παρέχει οριζόντια ορατότητα μέσω των εννοιών:

**Exposure (Έκθεση)**: Συνολική οικονομική έκθεση/άνοιγμα.
**Overdue (Ληξιπρόθεσμα)**: Εκκρεμότητες πέραν της προθεσμίας.
**Upcoming (Προσεχή)**: Αναμενόμενες ροές στο άμεσο μέλλον.

Σημείωση: Το Monitoring Layer δεν παράγει αυτόνομη πληροφορία· αντανακλά την κατάσταση των αντικειμένων στους κύκλους Revenue και Spend.

Οι λεπτομέρειες του revenue flow αποτυπώνονται στο `00A` και στα module documents.

## 4. Τι μένει εκτός scope

Το v1 παραμένει εστιασμένο σε λειτουργίες καθημερινής οικονομικής διαχείρισης.
Για τη διατήρηση της λειτουργικής εστίασης, το v1 **δεν** περιλαμβάνει:
- Γενική Λογιστική (General Ledger) και λογιστικές εγγραφές.
- Μηχανή φορολογικής και νομικής συμμόρφωσης (Compliance).
- Τραπεζική διασύνδεση ως πρωτογενή πηγή αλήθειας (Reconciliation). 
- Πλήρες σύστημα προμηθειών (Procurement depth).
- Προβλέψεις (Forecasting).

## 5. Βασική φιλοσοφία

Το σύστημα σχεδιάζεται με μερικές απλές αρχές.

### Πρώτα η έννοια, μετά η υλοποίηση

1. **Εννοιολογική Προτεραιότητα**: Η επιχειρησιακή έννοια προηγείται του σχεδιασμού της οθόνης ή του κώδικα.
2. **Ορολογική Συνοχή**: Μία λέξη φέρει την ίδια ακριβώς σημασία σε όλα τα modules.
3. **Παράγωγη Πληροφορία**: Η εικόνα ελέγχου (Monitoring) προκύπτει αποκλειστικά από τα βασικά αντικείμενα (Objects).
4. **Διαχωρισμός Ρόλων**: Το Overview (Επισκόπηση) λειτουργεί ως κέλυφος παρακολούθησης και όχι ως χώρος εκτέλεσης ενεργειών.

## 6. Λειτουργική Λογική Κύκλων

### Revenue Loop (Έσοδα)

Το `Invoice` γεννά issued truth (snapshot), από την οποία προκύπτει το `Receivable`. Η είσπραξη ολοκληρώνει το loop ως `Incoming Payment`.

Η ακριβής semantic ερμηνεία του `Issue` και οι κανόνες derivation/totals alignment ορίζονται στο `00A — Finance Domain Model & System Alignment`.

## 7. Spend Loop (Δαπάνες)

Ο κύκλος δαπανών ακολουθεί την αλυσίδα:

`Purchase Request -> Commitment -> Supplier Bill -> Outgoing Payment`

Οι canonical κανόνες για anti-overlap/relief (ώστε να αποφεύγεται διπλομέτρηση μεταξύ `Commitment` και `Supplier Bill`) ορίζονται στο `00A — Finance Domain Model & System Alignment`, ενώ η module-δομή και οι εξαρτήσεις αποτυπώνονται στο `01 — Finance Module Map`.

## 8. Monitoring and Control Layer

Το Monitoring and Control Layer παρέχει συνολική εικόνα, προτεραιοποίηση και σήματα προσοχής πάνω στα operational outputs του συστήματος.

Αποτελεί layer υπολογιζόμενων views και signals, όχι ανεξάρτητη πηγή οικονομικής αλήθειας.  
Η ακριβής semantic ερμηνεία των monitoring concepts και των ownership boundaries ορίζεται στο `00A — Finance Domain Model & System Alignment`.

## 9. Επισκόπηση modules

Σε επίπεδο brief, το σύστημα χωρίζεται σε τρεις βασικές περιοχές:
- Revenue λειτουργίες
- Spend λειτουργίες
- Monitoring / Control λειτουργίες

Τα modules οργανώνονται γύρω από τα βασικά objects του συστήματος και όχι γύρω από διαφορετικές ή αντικρουόμενες ερμηνείες της οικονομικής κατάστασης.

Η αναλυτική χαρτογράφηση ρόλων, ownership και σχέσεων γίνεται στο `01 — Finance Module Map`, με υποχρεωτική συμβατότητα προς το `00A`.

## 10. Θεμελιώδεις Έννοιες (Glossary)



Ο παρακάτω πίνακας λειτουργεί ως συνοπτικό semantic anchor για βασικούς όρους του v1.

| Έννοια | Περιγραφή |
| --- | --- |
| Overview | Monitoring shell, όχι execution workspace. |
| Invoice | Issued document truth στο revenue side. |
| Receivable | Απαίτηση που παράγεται από issued invoice truth. |
| Commitment | Upstream εγκεκριμένη δέσμευση δαπάνης πριν από `Supplier Bill`. |
| Supplier Bill | Supplier-side obligation truth στο spend side. |
| Exposure | Computed monitoring concept για συνολική έκθεση/άνοιγμα. |
| Overdue | Computed signal καθυστέρησης βάσει due date. |
| Upcoming | Computed signal προσεχών ροών/υποχρεώσεων. |

## 11. Αρχιτεκτονική Δομή Εγγράφων

Το documentation set ακολουθεί την εξής λογική:  
`00` ορίζει purpose και scope, `00A` ορίζει canonical meaning και rules, `01` ορίζει module structure, και τα `02+` εξειδικεύουν τα επιμέρους modules και workflows.

Όλα τα downstream έγγραφα οφείλουν να είναι συμβατά με τα `00` και `00A`.

## 12. Συμπέρασμα

Το Finance System v1 δεν φιλοδοξεί να αντικαταστήσει το λογιστήριο, αλλά να αποτελέσει το λειτουργικό τιμόνι της επιχείρησης. Κάθε προσθήκη ή αλλαγή στο σύστημα πρέπει να είναι συμβατή με τις αρχές και τις έννοιες που ορίζονται σε αυτό το Brief.
